1.  The first is the block-type lifecycle and runtime contract. The doc shows manifests and client.js files, and references methods like init, render, setField, but it doesn't pin down what a block-type implementation actually has to provide. What functions/exports does client.js need? What does the runtime call on it, in what order, with what arguments? What's the shape of "local state" a block owns — is it a plain object the runtime hands the block, or does the block declare its state shape? What's the protocol for the block telling the runtime "this field changed" so binding and watchers fire? Without nailing this, your very first text-editor block is being designed and built simultaneously, which usually produces a runtime that accidentally encodes whatever the first block happened to need. Two paragraphs and a tiny example block-type definition would lock this down.

2.  The second is the pathResolver.js algorithm — not the rules (those are now clean), but the actual resolution procedure. Given a reference string like {^projectRecord.budget} written inside block X, what does the resolver do step by step? Where does it get the "tree of named blocks"? Is that tree built at app load and held in memory? How does it know X's ancestor chain — does every block carry a pointer to its parent, or does the resolver walk a single source-of-truth tree? Is resolution done once at load (compile-time binding) or on every read (runtime lookup)? This last question is actually a fork in the road: compile-time binding gives you load-time errors and fast reads but makes dynamic block insertion harder; runtime lookup is more flexible but moves errors later and costs per-read work. You probably want compile-time, given the philosophy, but the doc doesn't say.

3.  The third is the watcher execution model. Watchers are mentioned throughout and are clearly central, but their semantics aren't pinned: when a field changes, in what order do dependent watchers fire? Synchronously or queued? What happens if watcher A's body modifies a field that watcher B watches — does B fire in the same tick, the next tick, or is that an error (cycle detection)? Can a watcher see the old value as well as the new? Reactive systems live or die on these answers, and getting them wrong on day one tends to manifest as subtle race-condition bugs months later that are almost impossible to untangle because every block in the app depends on whatever behavior accidentally emerged.

4.  The fourth is the first vertical slice spec, concretely. The doc says "start with Knowledge Foyer's simplest path" but doesn't say which path, in what order, with what acceptance criteria. "Author drafts an article, saves, sees it persisted, reloads page and sees it back" is a fine first slice and exercises Wire/REST + binding + autosave + path resolution + one block-type. Pin the slice, write 4–6 acceptance bullets, and you have a target that tells you when you're done with v0.1 instead of "build until it feels framework-shaped."

The renderer-neutral split (HTML/CSS/JS now, other targets later — defer the abstraction until you have two renderers wanting it), the MCP façade itself (decide later; just don't paint into the namespace), the remaining domain-syntax decision-points (quoting, regex form, two-value widget default — these don't block anything structural), inline-JS-vs-separated, and most of WorkSplicer-specific concerns. These are all decisions you can make better with one block-type and one slice running in front of you than you can in the abstract.

6.  How does a manifest.json look concretely (one example, ~10 lines, covering: declared parameters, declared state, declared ^name ancestor expectations if any, lifecycle exports)? And: when a block instance in app.xml writes name="{.name}" to bind a parameter, what is the exact resolution of that expression — does it evaluate in the parent's context (yes, almost certainly) and pass the result in, or is the binding kept live so changes to the parent's .name propagate to the child's local name? The latter is what you probably want (otherwise parameters are one-shot copies, which isn't reactive), but the doc doesn't say.
